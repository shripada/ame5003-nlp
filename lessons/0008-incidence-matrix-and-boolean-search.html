Searching a collection

The term-document incidence matrix, and answering Boolean queries with it — a clean idea that works, right up until it does not fit in memory.

A change of gear

Everything so far — tokenizing, tagging, finding names — worked on one piece of text. Now the question changes: given a whole collection of documents, which ones match what I am looking for? This is information retrieval, and it is what a search engine does.

Start with the simplest useful kind of search: Boolean queries. You ask for documents using AND, OR and NOT — for example, over the plays of Shakespeare:

Brutus AND Caesar AND NOT Calpurnia

"Find the plays that contain both Brutus and Caesar, but not Calpurnia." How would you answer that over a large collection, quickly?

The obvious way, and why it is not enough

You could scan every document for the words, like Ctrl-F or the Unix grep tool. For one file that is fine. For a big collection it is far too slow — you re-read every document on every query. And it does not handle NOT or complex combinations cleanly, nor let you rank results later. We need the collection organised in advance so queries are fast.

The term-document incidence matrix

Here is the first idea. Build a big table. One row per term (word), one column per document. Put a 1 where the term appears in that document, and a 0 where it does not.

For a handful of terms across six Shakespeare plays, it looks like this (1 = "the play contains this word"):

term Antony &
Cleopatra		Julius
Caesar		The
Tempest		 HamletOthelloMacbeth
Antony 110001
Brutus 110100
Caesar 110111
Calpurnia 010000
Cleopatra 100000
mercy 101111
worser 101110

Each row is now a string of 0s and 1s — a bit vector — that says exactly which plays contain that word. Read the Caesar row: 1 1 0 1 1 1 — it appears in every play except The Tempest.

Answering the query with bitwise operations

Once each term is a row of bits, a Boolean query is just bitwise arithmetic on those rows. Take the query back:

Brutus AND Caesar AND NOT Calpurnia

Pull out the three rows and combine them column by column. AND keeps a 1 only where both have a 1; NOT flips every bit:

Brutus            1 1 0 1 0 0
Caesar            1 1 0 1 1 1
NOT Calpurnia     1 0 1 1 1 1     ← Calpurnia was 0 1 0 0 0 0, flipped

Brutus AND Caesar             →   1 1 0 1 0 0
        AND NOT Calpurnia     →   1 0 0 1 0 0

The final vector is 1 0 0 1 0 0. The 1s are in columns 1 and 4, so the answer is:

Antony and Cleopatra     Hamlet

That is the whole method. Every Boolean query becomes a few bitwise operations on the rows of the matrix — and bitwise operations are about as fast as a computer gets. It is genuinely elegant.

Think about this before reading on

Using the matrix above, answer Antony AND mercy. Which plays come back?

Take the two rows and AND them column by column:

Antony   1 1 0 0 0 1
mercy    1 0 1 1 1 1
AND      1 0 0 0 0 1

1s in columns 1 and 6 → Antony and Cleopatra and Macbeth. Both words appear together only in those two plays.

Why this does not scale

The method is clean. The problem is the size of the table. Terms go down the side, documents across the top — so the matrix grows with both. Use the numbers from a realistic collection:

  • 1 million documents (columns)
  • about 500,000 distinct terms (rows)

That is 500,000 × 1,000,000 = 500 billion cells. Half a trillion 0s and 1s to store.

Now the killer detail. Each document contains maybe a thousand distinct words, so each column has about a thousand 1s and 499,000 zeros. Across the whole matrix there are at most about a billion 1s — which means:

The matrix is almost entirely zeros

Out of 500 billion cells, at most about 1 billion are 1s. That is 99.8% zeros.

You would be spending almost all of your memory storing the fact that words are absent from documents — which is nearly always true and almost never what you asked about. It will not fit, and even if it did, it would be a colossal waste.

So the incidence matrix is the right idea — organise the collection in advance so queries are fast — carried out the wrong way. The fix almost writes itself once you have seen the problem.

One more before the next session

Almost every cell in the matrix is a 0, and the 0s carry no information you care about. What is the obvious thing to change?

Stop storing the zeros. Record only the 1s.

For each term, instead of a full row of mostly-zero bits, just keep a list of the documents it actually appears in. That list is short, because a word is in few documents. No space wasted on absence.

That structure is called an inverted index, and it is the whole of session 9 — and the thing you will build in lab 4.

Question 1 of 2

In a term-document incidence matrix, what does a single row represent?

Correct answer: one term, as which documents it appears in.

Rows are terms, columns are documents. The Caesar row 1 1 0 1 1 1 says Caesar appears in every play but the third. A Boolean query then combines these rows with bitwise AND, OR and NOT.

Question 2 of 2

What is the main problem with the incidence matrix at real scale?

Correct answer: it is almost all zeros, so it wastes nearly all its space.

A million documents by half a million terms is 500 billion cells, of which at most a billion are 1s — about 99.8% zeros. The bitwise operations are fast; it is storing all that absence that sinks it. Recording only the 1s is the fix, and that is the inverted index.

The one thing to remember

The whole lesson in two sentences

A term-document incidence matrix turns Boolean search into fast bitwise operations on rows of 0s and 1s, which is elegant and correct. But at real scale it is 99.8% zeros and cannot fit, so the practical answer is to keep only the 1s — the inverted index of the next session.
